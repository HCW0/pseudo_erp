네이밍이라는 것은, 그렇게까지 로우레벨로 내려갈 필요는 없다.

단지 협업하는 입장끼리 통하는 정도면 됨.

예를 들어, php파일까지만 이름 규칙을 정하고 내부 변수,필드와 각 파일의 관계를

ERD로 표현할 수도 있을 것이다.



RFP

request for proposal

제안요청서


즉, 개발자에게 내줄 과제를 설명하는 문서를 말한다.

과제목표나 요구사항, 평가기준등이 들어가 잇다.

사용자나 고객의 의견이 적극반영되므로, 뭘만들어야할지

이해하기 쉽게 만들면된다.



WBS

work breakdown structure

작업 분해도


프로젝트의 범위랑 최종산출물을 세부요소로 분할해서 , 계층구조로 만든 시각자료

해당 과정에서 고려되어야할 것은 ERD~데이터 입력,처리,출력에서부터 어떤 모듈을 외부에서

수입해올 것인지, 분업 구조 및 하나하나의 흐름 그리고 검증(유닛 테스트)까지의 일을 말한다.

software development life cycle

SI, SM

System integration

	시스템 구축


	회사에서 시스템을 수주 받고, 고객이 갑이고

	회사에서 개발 용역을 굴리는데, 이 개발하는 친구들이 SI맨들임.




System management

	시스템 관리

	SI의 메뉴얼을 보고, 유지보수하는 것.



	ERD 

		모듈이 프로세스면, 테이블의 레코드는 데이터가 된다.


		ERD는 모듈과 데이터의 관계를 표현한 것을 말한다.





헤더와 디테일


	업무 특성상 상세 정보를 부를 수록 더 많은 필드를 가지게 된다. 존나 복잡하다.


	유에니, 최초에 로드할 레이블을 헤더라고 부르고 추가적으로 요구할 레이블을 상세라고 한다.


	상세에서 또 상세를 요구할 수 있기 때문에


	계층 구조, 깊이구조를 만들고 각 계층의 관계를 연결할 key set이 필요하다.


	




ERD  ~  Entity relationship diagram


	개체 관계 모델

	데이터 모델의 일종, 데이터 모델은 모델을 기술하기 위한 메타 데이터를 가진다.

	해당 모델에서 바라는 것은 데이터의 입력 프로세싱 출력 과정에 어떠한 테이블이

	관여할 것인지에 대해 테이블의 정의부터 입출력까지 획일화된 형식을 원한다.





모듈 설계서

	역시 인수인계할 유지보수팀이 읽어야하므로 형식이 중요하다.

	주로 그래픽 부분과 내부 로직으로 이분화시켜 작성하는게 이해하기 편할 것이다.

	내부에는 그래픽, 로직 외에도 검토사항이 필요한데,

	검토사항이란 해당 모듈이 구현해야하는 기능에 대한 것으로

	이게 없으면, 유지보수하는 과정에서 똑같은 기능을 추가시킬 수도 있으므로 작성해야한다.



	해당 과정에서 중요한 것은 얼마나 코딩이 잘 짜여졋는가 보다는

	앞으로의 유지보수에 있어서 얼마나 인수인계가 잘 될것인가 이다.


	리펙토링 같은 코드 개선 작업은 유지보수 파트이므로, 일단은

	사용하는데 지장이 없는 정도로 만드는 것을 생각하고


	
	설계서나 형식이 만들어지면, 거기에 차근차근 쌓아올리면 된다.
